= Réseau de stockage
:revdate: 2025-11-12
:page-revdate: {revdate}

{harvester-product-name} utilise {longhorn-product-name} pour fournir des volumes de périphériques de bloc pour les machines virtuelles et les pods. Si vous souhaitez isoler le trafic de réplication de {longhorn-product-name} du réseau de cluster intégré `mgmt` ou d'autres charges de travail à l'échelle du cluster, vous pouvez utiliser un réseau de stockage dédié pour une meilleure bande passante et performance réseau.

Pour plus d'informations, voir https://documentation.suse.com/cloudnative/storage/1.8/en/longhorn-system/networking/storage-network.html[Réseau de stockage] dans la documentation {longhorn-product-name}.

[NOTE]
====
Évitez de configurer directement les paramètres de {longhorn-product-name}, car cela peut entraîner un comportement système inattendu ou indésirable.
====

== Conditions préalables

Avant de commencer à configurer le réseau de stockage, assurez-vous que les exigences suivantes sont remplies{nbsp}:

* Les commutateurs réseau sont correctement configurés et un ID VLAN dédié est attribué au réseau de stockage.

* Le xref:networking/cluster-network.adoc[réseau de cluster] et le xref:networking/vm-network.adoc#_vlan_network[réseau VLAN] sont configurés correctement. Assurez-vous que les deux réseaux couvrent tous les nœuds et sont accessibles.

* La plage IP du réseau de stockage a les caractéristiques suivantes{nbsp}:
+
** Utilise le format CIDR IPv4
+
** N'entre pas en conflit et ne chevauche pas les réseaux de cluster Kubernetes
+
Les adresses suivantes sont réservées{nbsp}: `10.42.0.0/16`, `10.43.0.0/16`, `10.52.0.0/16` et `10.53.0.0/16`.
+
** Couvre les exigences du cluster
+
Le nombre requis d'adresses IP est calculé à l'aide de la formule suivante{nbsp}: `Required number of IPs = (Number of nodes * 2) + (Number of disks * 2) + Number of images to be downloaded or uploaded`
+
Exemple : Si un cluster a cinq nœuds avec deux disques chacun, et que dix images doivent être téléversées simultanément, la plage IP doit être supérieure ou égale à `/26` (calcul{nbsp}: (5 x 2) + (5 x 2) + 10 = 30).
+
** Exclut les adresses IP que {longhorn-product-name} les pods et le réseau de stockage ne doivent pas utiliser, telles que les adresses réservées pour xref:integrations/rancher/csi-driver.adoc#_rwx_volumes_support[volumes RWX], la passerelle et d'autres composants.

* Le CNI Whereabouts est installé correctement.
+
Vous pouvez vérifier si le CRD `ippools.whereabouts.cni.cncf.io` existe dans le cluster en utilisant la commande `kubectl get crd ippools.whereabouts.cni.cncf.io`.
+
Si une chaîne vide est renvoyée, ajoutez les CRDs dans https://github.com/harvester/harvester/tree/v1.1.0/deploy/charts/harvester/dependency_charts/whereabouts/crds[ce répertoire] en utilisant les commandes suivantes{nbsp}:
+
[,console]
----
kubectl apply -f https://raw.githubusercontent.com/harvester/harvester/v1.1.0/deploy/charts/harvester/dependency_charts/whereabouts/crds/whereabouts.cni.cncf.io_ippools.yaml
kubectl apply -f https://raw.githubusercontent.com/harvester/harvester/v1.1.0/deploy/charts/harvester/dependency_charts/whereabouts/crds/whereabouts.cni.cncf.io_overlappingrangeipreservations.yaml
----
+
[NOTE]
====
Le CNI Whereabouts n'est pas installé correctement dans certains https://github.com/harvester/harvester/issues/3168[scénarios de mise à niveau].
====

* Toutes les machines virtuelles sont arrêtées.
+
Vous pouvez vérifier l'état des machines virtuelles en utilisant la commande `kubectl get -A vmi`, qui devrait renvoyer une chaîne vide.
+
[CAUTION]
====
{harvester-product-name} envoie un signal d'arrêt gracieux aux machines virtuelles qui sont arrêtées via l'interface utilisateur {harvester-product-name}. Cependant, les charges de travail sont interrompues et restent indisponibles jusqu'à ce que vous démarriez manuellement les machines virtuelles après avoir confirmé que la configuration du réseau de stockage a été appliquée avec succès.
====

* Tous les pods attachés aux volumes {longhorn-product-name} sont arrêtés.

* Tous les téléchargements et téléversements d'images en cours sont soit terminés, soit supprimés.

== {longhorn-product-name} Routage du trafic de réplication

Le routage du {longhorn-product-name} trafic de réplication dépend de si le trafic VLAN des machines virtuelles et le {longhorn-product-name} réseau de stockage partagent les mêmes interfaces physiques ou utilisent des interfaces différentes.

* *Mêmes interfaces physiques* : Dans l'exemple suivant, `eth2` et `eth3` sont utilisés pour le trafic VLAN des machines virtuelles et le {longhorn-product-name} réseau de stockage. La ligne rouge indique que {longhorn-product-name} envoie le trafic de réplication via `eth3`.
+
image::storagenetwork/storagenetwork-same.png[storagenetwork-same.png]
+
[NOTE]
====
Vous devez inclure `eth2` et `eth3` dans la configuration du réseau de cluster et du réseau VLAN.
====

* *Interfaces physiques différentes* : Dans l'exemple suivant, `eth2` et `eth3` sont utilisés pour le trafic VLAN des machines virtuelles, tandis que `eth4` et `eth5` sont utilisés pour le réseau de stockage {longhorn-product-name}. La ligne rouge indique que {longhorn-product-name} envoie le trafic de réplication via `eth4`.
+
image::storagenetwork/storagenetwork-diff.png[storagenetwork-diff.png]
+
[NOTE]
====
Vous devez inclure `eth4` et `eth5` dans la configuration du réseau de cluster et du réseau VLAN.
====

== `storage-network` Paramètre

Le paramètre xref:../installation-setup/config/settings.adoc#_storage_network[`storage-network`] vous permet de configurer le réseau utilisé pour isoler le trafic de stockage intra-cluster lorsque la séparation est requise.

Vous pouvez activer et désactiver le réseau de stockage en utilisant soit l'interface utilisateur, soit l'interface en ligne de commande. Lorsque le paramètre est activé, vous devez construire un CRD Multus `NetworkAttachmentDefinition` en configurant certains champs.

Une fois le paramètre `storage-network` appliqué, {harvester-product-name} effectue les actions suivantes :

* Arrête tous les pods liés aux volumes {longhorn-product-name}, Prometheus, Grafana, Alertmanager et au contrôleur d'importation de VM.
* Crée un nouveau `NetworkAttachmentDefinition` et met à jour le {longhorn-product-name} paramètre de réseau de stockage.
* Redémarre tous les pods `instance-manager` et `backing-image-manager` pour appliquer la nouvelle configuration réseau.

=== Étapes de configuration

[tabs]
======
UI::
+
--
[TIP]
====
Il est fortement recommandé d'utiliser {harvester-product-name} l'interface utilisateur pour configurer le `storage-network` paramètre.
====

==== Activer le réseau de stockage

. Allez à *Paramètres avancés -> réseau de stockage ->*.

. Sélectionnez *Activé*.

. Configurez les champs *ID VLAN*, *Réseau de cluster*, *Plage IP* et *Exclure* pour construire un CRD Multus `NetworkAttachmentDefinition`.

. Cliquez sur *Enregistrer*.

image::storagenetwork/storage-network-enabled.png[Réseau de stockage activé]

==== Désactiver le réseau de stockage

. Allez à *Paramètres avancés -> réseau de stockage ->*.

. Sélectionnez *Désactiver*.

. Cliquez sur *Enregistrer*.

Une fois le réseau de stockage désactivé, {longhorn-product-name} commence à utiliser le réseau de pods pour les opérations liées au stockage.

image::storagenetwork/storage-network-disabled.png[Réseau de stockage désactivé]
--

CLI::
+
--
Vous pouvez utiliser la commande suivante pour configurer le paramètre xref:../installation-setup/config/settings.adoc#_storage_network[`storage-network`].

[,bash]
----
kubectl edit settings.harvesterhci.io storage-network
----

Le réseau de stockage est automatiquement activé dans les situations suivantes{nbsp}:

* Le champ `value` contient une chaîne JSON valide.
+
[,yaml]
----
apiVersion: harvesterhci.io/v1beta1
kind: Setting
metadata:
  name: storage-network
value: '{"vlan":100,"clusterNetwork":"storage","range":"192.168.0.0/24", "exclude":["192.168.0.100/32"]}'
----

* Le champ `value` est vide.
+
[,yaml]
----
apiVersion: harvesterhci.io/v1beta1
kind: Setting
metadata:
  name: storage-network
value: ''
----

Le réseau de stockage est désactivé lorsque vous retirez le champ `value`.

[,yaml]
----
apiVersion: harvesterhci.io/v1beta1
kind: Setting
metadata:
  name: storage-network
----

[CAUTION]
====
{harvester-product-name} considère les caractères insignifiants supplémentaires dans une chaîne JSON comme une configuration différente.
====
--

======

==== Changez la MTU du réseau de stockage

Voir xref:./cluster-network.adoc#_change_the_mtu_of_a_network_configuration_with_an_attached_storage_network[Changer la MTU d'une configuration réseau avec un réseau de stockage attaché].

. Testez la communication entre les {longhorn-product-name} pods.
+
Le réseau de stockage est dédié à la communication interne entre les {longhorn-product-name} pods, ce qui entraîne des performances et une fiabilité élevées. Cependant, le réseau de stockage dépend toujours de l'xref:./deep-dive.adoc#_external_networking[infrastructure réseau externe] pour la connectivité (similaire à la façon dont le xref:./vm-network.adoc#_create_a_vm_with_vlan_network[réseau VLAN des VM] fonctionne). Lorsque le réseau externe n'est pas connecté et configuré correctement, vous pouvez rencontrer les problèmes suivants{nbsp}:
+
* La machine virtuelle nouvellement créée reste bloquée à l'état `Not-Ready`.
* Les journaux du pod `longhorn-manager` contiennent des messages d'erreur.
+

+

+

+

+
Le réseau de stockage peut mal fonctionner en raison de problèmes avec le réseau externe, tels que les suivants{nbsp}:
+
** Les NIC physiques (installés sur {harvester-product-name} nœuds) associés au réseau de stockage n'ont pas été ajoutés au même VLAN dans les commutateurs externes.
** Les commutateurs externes ne sont pas correctement connectés et configurés.

Une fois la configuration vérifiée, vous pouvez démarrer manuellement des machines virtuelles si nécessaire.

== Meilleures pratiques

* Lors de la configuration d'une plage d'adresses IP pour le réseau de stockage, assurez-vous que les adresses IP allouées peuvent répondre aux besoins futurs du cluster. C'est important car les pods {longhorn-product-name} (`instance-manager` et `backing-image-manager`) cessent de fonctionner lorsque de nouveaux nœuds sont ajoutés au cluster ou que plus de disques sont ajoutés à un nœud après la configuration du réseau de stockage, et lorsque le nombre requis d'adresses IP dépasse les adresses IP allouées. Résoudre le problème implique de reconfigurer le réseau de stockage avec la plage d'adresses IP correcte.
+
{longhorn-product-name} pods utilisent le réseau de stockage comme suit{nbsp}:
+
** `instance-manager` pods{nbsp}: Les composants du gestionnaire d'instances ont été https://longhorn.io/docs/1.5.0/deploy/important-notes/#instance-managers-consolidated[consolidés dans {longhorn-product-name} v1.5.0]. Chaque nœud nécessite une adresse IP. Lors d'une mise à niveau, les anciennes et nouvelles versions de ces pods existent, et l'ancienne version est supprimée une fois la mise à niveau terminée.
+
** `backing-image-ds` pods{nbsp}: Ces pods traitent les téléversements et les téléchargements d'images de données sources à la volée, et sont supprimés une fois que les téléversements et les téléchargements d'images sont terminés.
+
** `backing-image-manager` pods{nbsp}: Chaque disque nécessite une adresse IP. Lors d'une mise à niveau, les anciennes et nouvelles versions de ces pods existent, et l'ancienne version est supprimée une fois la mise à niveau terminée.

* Configurez le réseau de stockage sur un réseau de cluster non-`mgmt` pour garantir une séparation complète du trafic de réplication {longhorn-product-name} du trafic du plan de contrôle Kubernetes. Utiliser `mgmt` est possible mais pas recommandé en raison de l'impact négatif (concurrence des ressources et de la bande passante) sur les performances du réseau du plan de contrôle. Utilisez `mgmt` uniquement si votre cluster a des contraintes liées aux NIC et si vous pouvez complètement séparer le trafic.
