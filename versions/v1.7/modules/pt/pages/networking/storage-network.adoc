= Rede de Armazenamento
:revdate: 2025-11-12
:page-revdate: {revdate}

{harvester-product-name} utiliza {longhorn-product-name} para fornecer volumes de dispositivos de bloco para máquinas virtuais e pods. Se você deseja isolar o tráfego de replicação de {longhorn-product-name} da `mgmt` (a rede de cluster incorporada) ou de outras cargas de trabalho em todo o cluster, pode usar uma rede de armazenamento dedicada para melhor largura de banda e desempenho da rede.

Para mais informações, consulte https://documentation.suse.com/cloudnative/storage/1.8/en/longhorn-system/networking/storage-network.html[Rede de Armazenamento] na documentação de {longhorn-product-name}.

[NOTE]
====
Evite configurar as configurações de {longhorn-product-name} diretamente, pois isso pode resultar em comportamento inesperado ou indesejado do sistema.
====

== Pré-requisitos

Antes de começar a configurar a rede de armazenamento, certifique-se de que os seguintes requisitos sejam atendidos:

* Os switches de rede estão configurados corretamente e um ID de VLAN dedicado é atribuído à rede de armazenamento.

* A xref:networking/cluster-network.adoc[rede de cluster] e a xref:networking/vm-network.adoc#_vlan_network[rede VLAN] estão configuradas corretamente. Certifique-se de que ambas as redes cobrem todos os nós e são acessíveis.

* O intervalo de IP da rede de armazenamento possui as seguintes características:
+
** Utiliza o formato CIDR IPv4
+
** Não conflita ou se sobrepõe às redes de cluster do Kubernetes
+
Os seguintes endereços estão reservados: `10.42.0.0/16`, `10.43.0.0/16`, `10.52.0.0/16` e `10.53.0.0/16`.
+
** Atende aos requisitos do cluster
+
O número necessário de endereços IP é calculado usando a seguinte fórmula: `Required number of IPs = (Number of nodes * 2) + (Number of disks * 2) + Number of images to be downloaded or uploaded`
+
Exemplo: Se um cluster tem cinco nós com dois discos cada, e dez imagens devem ser carregadas simultaneamente, o intervalo de IP deve ser maior ou igual a `/26` (cálculo: (5 x 2) + (5 x 2) + 10 = 30).
+
** Exclui endereços IP que {longhorn-product-name} pods e a rede de armazenamento não devem usar, como endereços reservados para xref:integrations/rancher/csi-driver.adoc#_rwx_volumes_support[volumes RWX], o gateway e outros componentes.

* O CNI Whereabouts está instalado corretamente.
+
Você pode verificar se o `ippools.whereabouts.cni.cncf.io` CRD existe no cluster usando o comando `kubectl get crd ippools.whereabouts.cni.cncf.io`.
+
Se uma string vazia for retornada, adicione os CRDs em https://github.com/harvester/harvester/tree/v1.1.0/deploy/charts/harvester/dependency_charts/whereabouts/crds[este diretório] usando os seguintes comandos:
+
[,console]
----
kubectl apply -f https://raw.githubusercontent.com/harvester/harvester/v1.1.0/deploy/charts/harvester/dependency_charts/whereabouts/crds/whereabouts.cni.cncf.io_ippools.yaml
kubectl apply -f https://raw.githubusercontent.com/harvester/harvester/v1.1.0/deploy/charts/harvester/dependency_charts/whereabouts/crds/whereabouts.cni.cncf.io_overlappingrangeipreservations.yaml
----
+
[NOTE]
====
O CNI Whereabouts não está instalado corretamente em certos https://github.com/harvester/harvester/issues/3168[cenários de fazer upgrade].
====

* Todas as máquinas virtuais estão paradas.
+
Você pode verificar o status das máquinas virtuais usando o comando `kubectl get -A vmi`, que deve retornar uma string vazia.
+
[CAUTION]
====
{harvester-product-name} envia um sinal de desligamento gracioso para as máquinas virtuais que estão paradas usando a interface {harvester-product-name}. No entanto, as cargas de trabalho são interrompidas e permanecem indisponíveis até que você inicie manualmente as máquinas virtuais após confirmar que a configuração da rede de armazenamento foi aplicada com sucesso.
====

* Todos os pods que estão anexados a {longhorn-product-name} volumes estão parados.

* Todos os uploads e downloads de imagens em andamento estão completos ou foram excluídos.

== {longhorn-product-name} Roteamento de tráfego de replicação

O roteamento do {longhorn-product-name} tráfego de replicação depende de se o tráfego VLAN da máquina virtual e a {longhorn-product-name} rede de armazenamento compartilham as mesmas interfaces físicas ou usam interfaces diferentes.

* *Mesmas interfaces físicas*: No exemplo a seguir, tanto `eth2` quanto `eth3` são usados para o tráfego VLAN da máquina virtual e a {longhorn-product-name} rede de armazenamento. A linha vermelha indica que {longhorn-product-name} envia tráfego de replicação através de `eth3`.
+
image::storagenetwork/storagenetwork-same.png[storagenetwork-same.png]
+
[NOTE]
====
Você deve incluir `eth2` e `eth3` na configuração da rede do cluster e da rede VLAN.
====

* *Interfaces físicas diferentes*: No exemplo a seguir, `eth2` e `eth3` são usados para o tráfego VLAN da máquina virtual, enquanto `eth4` e `eth5` são usados para a rede de armazenamento {longhorn-product-name}. A linha vermelha indica que {longhorn-product-name} envia tráfego de replicação através de `eth4`.
+
image::storagenetwork/storagenetwork-diff.png[storagenetwork-diff.png]
+
[NOTE]
====
Você deve incluir `eth4` e `eth5` na configuração da rede do cluster e da rede VLAN.
====

== `storage-network` Configuração

A configuração xref:../installation-setup/config/settings.adoc#_storage_network[`storage-network`] permite que você configure a rede usada para isolar o tráfego de armazenamento em cluster quando a segregação é necessária.

Você pode habilitar e desabilitar a rede de armazenamento usando a interface ou a CLI. Quando a configuração está habilitada, você deve construir um CRD Multus `NetworkAttachmentDefinition` configurando certos campos.

Uma vez que a configuração `storage-network` é aplicada, {harvester-product-name} realiza as seguintes ações:

* Interrompe todos os pods relacionados a {longhorn-product-name} volumes, Prometheus, Grafana, Alertmanager e o Controlador de Importação de VM.
* Cria um novo `NetworkAttachmentDefinition` e atualiza a {longhorn-product-name} configuração de rede de armazenamento.
* Reinicia todos os pods `instance-manager` e `backing-image-manager` para aplicar a nova configuração de rede.

=== Passos de configuração

[tabs]
======
UI::
+
--
[TIP]
====
É fortemente recomendado usar a {harvester-product-name} interface para configurar o `storage-network`.
====

==== Habilitar a rede de armazenamento

. Vá para *Configurações Avançadas -> -> rede-de-armazenamento*.

. Selecione *Habilitado*.

. Configure os campos *VLAN ID*, *Rede do Cluster*, *Faixa de IP* e *Excluir* para construir um CRD Multus `NetworkAttachmentDefinition`.

. Clique em *Salvar*.

image::storagenetwork/storage-network-enabled.png[Rede de armazenamento habilitada]

==== Desabilitar a rede de armazenamento

. Vá para *Configurações Avançadas -> -> rede-de-armazenamento*.

. Selecione *Desabilitar*.

. Clique em *Salvar*.

Uma vez que a rede de armazenamento é desabilitada, {longhorn-product-name} começa a usar a rede de pods para operações relacionadas ao armazenamento.

image::storagenetwork/storage-network-disabled.png[Rede de armazenamento desabilitada]
--

CLI::
+
--
Você pode usar o seguinte comando para configurar o xref:../installation-setup/config/settings.adoc#_storage_network[`storage-network`].

[,bash]
----
kubectl edit settings.harvesterhci.io storage-network
----

A rede de armazenamento é automaticamente habilitada nas seguintes situações:

* O campo `value` contém uma string JSON válida.
+
[,yaml]
----
apiVersion: harvesterhci.io/v1beta1
kind: Setting
metadata:
  name: storage-network
value: '{"vlan":100,"clusterNetwork":"storage","range":"192.168.0.0/24", "exclude":["192.168.0.100/32"]}'
----

* O campo `value` está vazio.
+
[,yaml]
----
apiVersion: harvesterhci.io/v1beta1
kind: Setting
metadata:
  name: storage-network
value: ''
----

A rede de armazenamento é desabilitada quando você remover o campo `value`.

[,yaml]
----
apiVersion: harvesterhci.io/v1beta1
kind: Setting
metadata:
  name: storage-network
----

[CAUTION]
====
{harvester-product-name} considera caracteres insignificantes extras em uma string JSON como uma configuração diferente.
====
--

======

==== Altere o MTU da rede de armazenamento

Veja xref:./cluster-network.adoc#_change_the_mtu_of_a_network_configuration_with_an_attached_storage_network[Altere o MTU de uma configuração de rede com uma rede de armazenamento anexada].

** Os switches externos não estão conectados e configurados corretamente.

Uma vez que a configuração é verificada, você pode iniciar máquinas virtuais manualmente quando necessário.

== Melhores práticas

* Ao configurar um intervalo de IP para a rede de armazenamento, certifique-se de que os endereços IP alocados possam atender às necessidades futuras do cluster. Isso é importante porque os {longhorn-product-name} pods (`instance-manager` e `backing-image-manager`) param de funcionar quando novos nós são adicionados ao cluster ou mais discos são adicionados a um nó após a configuração da rede de armazenamento, e quando o número necessário de IPs excede os IPs alocados. Resolver o problema envolve reconfigurar a rede de armazenamento com o intervalo de IP correto.
+
{longhorn-product-name} pods usam a rede de armazenamento da seguinte forma:
+
** `instance-manager` pods: Os componentes do Instance Manager foram https://longhorn.io/docs/1.5.0/deploy/important-notes/#instance-managers-consolidated[consolidados na {longhorn-product-name} v1.5.0]. Cada nó requer um endereço IP. Durante o fazer upgrade, tanto as versões antigas quanto as novas desses pods existem, e a versão antiga é excluída assim que o fazer upgrade é concluído.
+
** `backing-image-ds` pods: Esses pods processam uploads e downloads de fontes de dados de imagem em tempo real e são removidos assim que os uploads e downloads de imagem são concluídos.
+
** `backing-image-manager` pods: Cada disco requer um endereço IP. Durante o fazer upgrade, tanto as versões antigas quanto as novas desses pods existem, e a versão antiga é excluída assim que o fazer upgrade é concluído.

* Configure a rede de armazenamento em uma rede de cluster não-`mgmt` para garantir a separação completa do tráfego de replicação {longhorn-product-name} do tráfego do plano de controle do Kubernetes. Usar `mgmt` é possível, mas não recomendado devido ao impacto negativo (conflito de recursos e largura de banda) no desempenho da rede do plano de controle. Use `mgmt` apenas se seu cluster tiver restrições relacionadas a NIC e se você puder segregar completamente o tráfego.
