= Upgrades
:revdate: 2026-01-02
:page-revdate: {revdate}

{harvester-product-name} está adotando uma nova estratégia de ciclo de vida que simplifica a gestão de versões e o upgrade. Essa estratégia inclui o seguinte:

* Ciclo de lançamento menor de quatro meses
* Ciclo de lançamento de patch de dois meses
* Política de adoção de componentes

[NOTE]
====
{harvester-product-name} não suporta downgrades. Essa restrição ajuda a prevenir comportamentos inesperados do sistema e problemas associados à incompatibilidade de funções, descontinuação e remoção.
====

* Atualizar de uma versão menor para a próxima (por exemplo, de v1.5.2 para v1.6.1) sem precisar instalar os patches lançados entre as duas versões. Isso é possível porque {harvester-product-name} permite um máximo de uma atualização de versão menor para componentes subjacentes.

== Gerenciamento de Máquinas Virtuais através do Fazer Upgrade

=== Máquinas Virtuais Migráveis ao Vivo

xref:virtual-machines/live-migration.adoc#_live_migratable_virtual_machines[Máquinas virtuais migráveis ao vivo] são automaticamente migradas para outros nós via xref:../virtual-machines/live-migration.adoc#_automatically_triggered_batch_migration[migração em lote] antes que o nó atual seja atualizado. Essas máquinas virtuais não experienciam tempo de inatividade durante a migração.

=== Máquinas Virtuais Não Migráveis

Quando um upgrade é acionado, {harvester-product-name} realiza certas ações dependendo do valor da opção xref:installation-setup/config/settings.adoc#_upgrade_config[`upgrade-config`] da configuração `restoreVM`.

* `false`: {harvester-product-name} não realiza o upgrade quando xref:virtual-machines/live-migration.adoc#_non_migratable_virtual_machines[máquinas virtuais não migráveis] ainda estão em execução. Você deve desligar manualmente as máquinas virtuais.

* `true`: {harvester-product-name} desliga automaticamente as máquinas virtuais não migráveis quando o nó é atualizado e, em seguida, as restaura após o nó ser reiniciado.

[CAUTION]
====
Máquinas virtuais não migráveis experienciam tempo de inatividade durante a migração.
====

Para mais informações, consulte xref:upgrades/troubleshooting.adoc#_phase_4_upgrade_nodes[Fase 4: Atualizar nós].

== Antes de iniciar um upgrade

Verifique a configuração disponível xref:../installation-setup/config/settings.adoc#_upgrade_config[`upgrade-config` para] ajustar as estratégias e comportamentos de atualização que melhor se adequam ao seu ambiente de cluster.

== Inicie um upgrade

[CAUTION]
====
* Antes de fazer upgrade do seu cluster {harvester-product-name}, recomendamos fortemente:
 ** Faça backup de suas VMs, se necessário.
* Não opere o cluster durante um upgrade. Por exemplo, criar novas VMs, fazer upload de novas imagens, etc., durante um upgrade.
* Certifique-se de que seu hardware atende aos requisitos de *hardware preferido* xref:../installation-setup/requirements.adoc#_hardware_requirements[hardware]. Isso se deve ao fato de que haverá recursos intermediários consumidos por uma atualização.
* Certifique-se de que cada nó tenha pelo menos 30 GiB de espaço livre na partição do sistema (`df -h /usr/local/`). Se algum nó no cluster tiver menos de 30 GiB de espaço livre na partição do sistema, a atualização será negada. Verifique o <<Free system partition space requirement,requisito de espaço livre na partição do sistema>> para mais informações.
* Execute o script de verificação prévia em um nó de plano de controle {harvester-product-name}. Por favor, escolha um script de acordo com a versão do seu cluster: https://github.com/harvester/upgrade-helpers/tree/main/pre-check.
* Um número de pods privilegiados temporários será criado nos namespaces `harvester-system` e `cattle-system` para realizar operações de atualização em nível de host. Se https://kubernetes.io/docs/concepts/security/pod-security-admission/[admissão de segurança de pod] estiver habilitada, ajuste essas políticas para permitir que esses pods sejam executados.
====

[CAUTION]
====
* Certifique-se de que os horários de todos os nós estejam sincronizados. Recomenda-se usar um servidor NTP para sincronizar o tempo. Se um servidor NTP não estiver configurado durante a instalação, você pode adicionar manualmente um servidor NTP *em cada nó*:
+
[,sh]
----
  $ sudo -i

  # Add time servers
  $ vim /etc/systemd/timesyncd.conf
  [ntp]
  NTP=0.pool.ntp.org

  # Enable and start the systemd-timesyncd
  $ timedatectl set-ntp true

  # Check status
  $ sudo timedatectl status
----
====

[CAUTION]
====
As NICs que se conectam a um barramento PCI podem ser renomeadas após uma atualização. Por favor, verifique o https://harvesterhci.io/kb/nic-naming-scheme[artigo da base de conhecimento] para mais informações.
====

[NOTE]
====
A partir da versão v1.7.0, {harvester-product-name} utiliza um repositório de atualização baseado em implantação em vez de uma abordagem baseada em máquina virtual para melhorar o desempenho e a confiabilidade. Para mais informações, consulte https://github.com/harvester/harvester/issues/7101[o problema #7101].
====

=== Personalize a versão

. Baixe o arquivo da versão (`pass:[https://releases.rancher.com/harvester/{version}/version.yaml]`).
+
Exemplo:
+
O arquivo da versão https://releases.rancher.com/harvester/v1.5.0/version.yaml[v1.5.0] é baixado como `v1.5.0.yaml`.

[,yaml]
----

  name: v1.5.0-customized # Changed, to avoid duplicated with the official version name
  namespace: harvester-system
spec:
  isoChecksum: 'df28e9bf8dc561c5c26dee535046117906581296d633eb2988e4f68390a281b6856a5a0bd2e4b5b988c695a53d0fc86e4e3965f19957682b74317109b1d2fe32'  # Don't change

  releaseDate: '20250425'
----

=== Prepare o arquivo ISO

. Baixe um arquivo ISO da página https://github.com/harvester/harvester/releases[Releases].

. Salve o ISO em um servidor HTTP local.
+
Assuma que o arquivo está hospedado em `http://10.10.0.1/harvester.iso`.

=== Prepare a Versão

. Baixe o arquivo da versão (`pass:[https://releases.rancher.com/harvester/{version}/version.yaml]`).

. Substitua o valor `isoURL` no arquivo.
+
[,yaml]
----
  apiVersion: harvesterhci.io/v1beta1
  kind: Version
  metadata:
    name: v1.5.0
    namespace: harvester-system
  spec:
    isoChecksum: <SHA-512 checksum of the ISO>
    isoURL: http://10.10.0.1/harvester.iso  # change to local ISO URL
    releaseDate: '20250425'
----
+
Assuma que o arquivo está hospedado em `http://10.10.0.1/version.yaml`. Se você precisar de personalizações, veja <<Customize the version>>.

. Acesse um dos nós do plano de controle via SSH e faça login usando a conta root.

. Crie um objeto de versão.
+
[,console]
----
rancher@node1:~> sudo -i
rancher@node1:~> kubectl create -f http://10.10.0.1/version.yaml
----

== Personalize as atualizações dos nós

As atualizações {harvester-product-name} envolvem várias fases definidas. Uma fase chave é a atualização dos nós, durante a qual o sistema operacional e a distribuição subjacente do Kubernetes (RKE2) são atualizados em cada nó sequencialmente e de forma autônoma.

Você tem a opção de pausar atualizações automáticas em nós específicos, útil para tarefas de manutenção ou verificação manuais. Após a conclusão dessas tarefas, você deve instruir explicitamente {harvester-product-name} a retomar o upgrade nos nós-alvo.

=== Pausando atualizações dos nós

Você pode usar a opção `nodeUpgradeOption` na configuração xref:installation-setup/config/settings.adoc#_upgrade_config[`upgrade-config`] para pausar as atualizações dos nós.

* Pausar para todos os nós no cluster: Altere o valor do campo `mode` para `manual`.
* Pausa para nós específicos: Liste os nomes dos nós no campo `pauseNodes`. Nós não incluídos na lista são automaticamente atualizados.

[IMPORTANT]
====
{harvester-product-name} aplica a configuração `nodeUpgradeOption` durante a fase de inicialização do upgrade. Mudanças feitas nesses campos após a inicialização são ignoradas para o upgrade atual e só terão efeito no próximo ciclo de upgrade.
====

[TIP]
====
Você pode modificar o recurso personalizado `Upgrade` para adicionar ou remover quaisquer nós na anotação, desde que o nó não tenha sido atualizado. A função de pausa da atualização do nó usa a anotação como fonte de verdade.
====

A interface do usuário {harvester-product-name} fornece confirmação visual das atualizações de nós pausadas. No exemplo a seguir, a atualização do nó `charlie-1-tink-system` está atualmente pausada.

image::upgrade/node-upgrade-paused.png[Atualização do nó pausada]

Você também pode usar o seguinte `kubectl` comando para verificar as atualizações de nós pausadas.

[,shell]
----
$ kubectl -n harvester-system get upgrades -l harvesterhci.io/latestUpgrade=true -o yaml
    ...
    annotations:
      harvesterhci.io/node-upgrade-pause-map: '{"charlie-1-tink-system":"pause","charlie-2-tink-system":"pause","charlie-3-tink-system":"pause"}'
    ...
    nodeStatuses:
      charlie-1-tink-system:
        message: Node upgrade paused as requested by the user
        reason: AdministrativelyPaused
        state: Node-upgrade paused
      charlie-2-tink-system:
        state: Images preloaded
      charlie-3-tink-system:
        state: Images preloaded
    ...
----

[CAUTION]
====
Os trabalhos de pré-drain para nós com atualizações pausadas não foram criados. No entanto, esses nós ainda estão isolados e você não poderá executar novas cargas de trabalho neles. Apenas tarefas de manutenção, como desligar manualmente máquinas virtuais, devem ser realizadas em nós com atualizações pausadas.
====

=== Retomando uma atualização de nó pausada

Você pode retomar uma atualização de nó pausada atualizando a anotação `harvesterhci.io/node-upgrade-pause-map` no recurso personalizado `Upgrade`.

Exemplo:

[,shell]
----
# Find out the latest Upgrade custom resource
$ kubectl -n harvester-system get upgrades -l harvesterhci.io/latestUpgrade=true
NAME                 AGE
hvst-upgrade-6mcwv   4h16m

# Update the annotation to unpause the node
$ kubectl -n harvester-system annotate --overwrite upgrades hvst-upgrade-6mcwv harvesterhci.io/node-upgrade-pause-map='{"charlie-1-tink-system":"unpause","charlie-2-tink-system":"pause","charlie-3-tink-system":"pause"}'
----

Uma vez que o nó de destino esteja anotado no recurso personalizado `Upgrade`, {harvester-product-name} retoma a atualização imediatamente, com a interface do usuário exibindo atualizações visuais de progresso.

image::upgrade/node-upgrade-unpaused.png[Atualização de nó pausada retomada]

Você também pode usar o seguinte `kubectl` comando para verificar o estado do nó de destino:

[,shell]
----
$ kubectl -n harvester-system get upgrades -l harvesterhci.io/latestUpgrade=true -o yaml
    ...
    annotations:
      harvesterhci.io/node-upgrade-pause-map: '{"charlie-1-tink-system":"unpause","charlie-2-tink-system":"pause","charlie-3-tink-system":"pause"}'
    ...
    nodeStatuses:
      charlie-1-tink-system:
        state: Pre-draining
      charlie-2-tink-system:
        state: Images preloaded
      charlie-3-tink-system:
        state: Images preloaded
    ...
----

Dependendo do número de nós de destino, pode ser necessário executar a operação de despausar várias vezes durante o processo geral de atualização do cluster.

== Requisito de espaço livre na partição do sistema

{harvester-product-name} carrega imagens em cada nó durante as atualizações. Quando o uso do disco excede o limite de coleta de lixo do kubelet, o kubelet exclui imagens não utilizadas para liberar espaço. Isso pode causar problemas em ambientes air-gapped, pois as imagens não estão disponíveis no nó.

{harvester-product-name} inclui verificações que garantem que os nós não acionem a coleta de lixo após carregar novas imagens.

Quando o espaço em disco é insuficiente, {harvester-product-name} bloqueia a atualização e retorna um erro semelhante ao seguinte:

[,console]
----
Node "harvester-node-0" will reach 92.84% storage space after loading new images. It's higher than kubelet image garbage collection threshold 85%.
----

Se você quiser tentar atualizar mesmo que o espaço livre na partição do sistema seja insuficiente em alguns nós, pode atualizar a anotação `harvesterhci.io/skipGarbageCollectionThresholdCheck: true` do objeto `Upgrade`.

[,yaml]
----
apiVersion: harvesterhci.io/v1beta1
kind: Upgrade
metadata:
  annotations:
    harvesterhci.io/skipGarbageCollectionThresholdCheck: true
  generateName: hvst-upgrade-
  namespace: harvester-system
spec:
  version: "1.6.0"
  logEnabled: true
----

[CAUTION]
====
Definir um valor menor do que o valor pré-definido pode causar a falha da atualização e não é recomendado em um ambiente de produção.
====

As seções a seguir descrevem soluções para problemas relacionados a esse requisito.

=== Liberar espaço na partição do sistema manualmente

{harvester-product-name} tenta remover imagens de contêiner desnecessárias após a conclusão da atualização. No entanto, essa limpeza automática de imagens pode não ser realizada por vários motivos. Você pode usar https://github.com/harvester/upgrade-helpers/blob/main/bin/harv-purge-images.sh[um script] para remover imagens manualmente. Para mais informações, consulte o problema https://github.com/harvester/harvester/issues/6620[#6620].

=== Configure um registro de contêiner privado e pule o pré-carregamento de imagens

A partição do sistema ainda pode não ter espaço livre mesmo após a remoção de imagens. Para resolver isso, configure um registro de contêiner privado para imagens atuais e novas, e configure a configuração xref:../installation-setup/config/settings.adoc#_upgrade_config[`upgrade-config`] com o seguinte valor:

[,json]
----
{"imagePreloadOption":{"strategy":{"type":"skip"}}, "restoreVM": false}
----

{harvester-product-name} pula o processo de pré-carregamento da imagem de atualização. Quando as implantações nos nós são atualizadas, o tempo de execução do contêiner carrega as imagens armazenadas no registro de contêiner privado.

[CAUTION]
====
Não confie no registro de contêiner público. Observe quaisquer interrupções potenciais no serviço de internet e quão próximo você está de atingir o https://www.docker.com/increase-rate-limits[limite de taxa do Docker Hub]. A falha ao baixar qualquer uma das imagens necessárias pode causar a falha da atualização e deixar o cluster em um estado intermediário.
====

== Verificação de expiração de certificados

{harvester-product-name} verifica o período de validade dos certificados em cada nó. Essa verificação elimina a possibilidade de os certificados expirarem enquanto a atualização está em andamento. Se um certificado expirar dentro de 7 dias, um erro é retornado. Esse comportamento pode ser substituído definindo a anotação `harvesterhci.io/minCertsExpirationInDay`.

Exemplo:

[,yaml]
----
apiVersion: harvesterhci.io/v1beta1
kind: Upgrade
metadata:
  annotations:
    harvesterhci.io/minCertsExpirationInDay: "14"
  generateName: hvst-upgrade-
  namespace: harvester-system
spec:
  version: "1.6.0"
  logEnabled: true
----

Quando essa anotação é adicionada ao objeto `Upgrade`, {harvester-product-name} retorna um erro quando detecta um certificado que expirará dentro de 14 dias.

Para mais informações, veja xref:installation-setup/config/settings.adoc#_auto_rotate_rke2_certs[auto-rotate-rke2-certs].

== Compatibilidade de backup de máquina virtual

Você pode encontrar certas limitações ao criar e restaurar backups que envolvem armazenamento externo.

== Falha do Longhorn Manager devido à evicção de imagem de apoio

[CAUTION]
====
Ao atualizar para {harvester-product-name} *v1.4.x*, o Longhorn Manager pode falhar se a flag `EvictionRequested` estiver definida como `true` em qualquer nó ou disco. Esse problema é causado por uma https://longhorn.io/kb/troubleshooting-longhorn-manager-crashes-due-to-backing-image-eviction/[condição de corrida] entre a exclusão de um disco na especificação da imagem de apoio e a atualização de seu status.

Para evitar que o problema ocorra, certifique-se de que a flag `EvictionRequested` esteja definida como `false` antes de iniciar o processo de atualização.
====

== Reative os webhooks de admissão do ingress-nginx do RKE2 (CVE-2025-1974)

Se você https://harvesterhci.io/kb/2025/03/25/cve-2025-1974[desativou os webhooks de admissão do ingress-nginx do RKE2] para mitigar https://nvd.nist.gov/vuln/detail/CVE-2025-1974[CVE-2025-1974], você deve reativar o webhook após atualizar para {harvester-product-name} v1.5.0 ou posterior.

. Verifique se {harvester-product-name} está usando nginx-ingress v1.12.1 ou posterior.
+
[,shell]
----
$ kubectl -n kube-system get po -l"app.kubernetes.io/name=rke2-ingress-nginx" -ojsonpath='{.items[].spec.containers[].image}'
rancher/nginx-ingress-controller:v1.12.1-hardened1
----

. Execute `kubectl -n kube-system edit helmchartconfig rke2-ingress-nginx` para *remover* as seguintes configurações do recurso `HelmChartConfig`.
+
* `.spec.valuesContent.controller.admissionWebhooks.enabled: false`
* `.spec.valuesContent.controller.extraArgs.enable-annotation-validation: true`

. Verifique se a nova configuração `.spec.ValuesContent` é semelhante ao exemplo a seguir.
+
[,yaml]
----
apiVersion: helm.cattle.io/v1
kind: HelmChartConfig
metadata:
  name: rke2-ingress-nginx
  namespace: kube-system
spec:
  valuesContent: |-
    controller:
      admissionWebhooks:
        port: 8444
      extraArgs:
        default-ssl-certificate: cattle-system/tls-rancher-internal
      config:
        proxy-body-size: "0"
        proxy-request-buffering: "off"
      publishService:
        pathOverride: kube-system/ingress-expose
----

=== A atualização está presa no estado "Pré-drenado".

O processo de atualização pode ficar preso no estado "Pré-drenado". O Kubernetes deve drenar a carga de trabalho no nó, mas alguns fatores podem causar a paralisação do processo.

image::upgrade/3730-stuck.png[]

Uma possível causa são processos relacionados a motores órfãos do Longhorn Instance Manager. Para determinar se isso se aplica à sua situação, execute os seguintes passos:

. Verifique o nome do pod `instance-manager` no nó preso.
+
Exemplo:
+
O nó preso é `harvester-node-1`, e o nome do pod do Longhorn Instance Manager é `instance-manager-d80e13f520e7b952f4b7593fc1883e2a`.
+
[,shell]
----
$ kubectl get pods -n longhorn-system --field-selector spec.nodeName=harvester-node-1 | grep instance-manager
instance-manager-d80e13f520e7b952f4b7593fc1883e2a          1/1     Running   0              3d8h
----

. Verifique os logs do Longhorn Manager em busca de mensagens informativas.
+
Exemplo:
+
[,shell]
----
$ kubectl -n longhorn-system logs daemonsets/longhorn-manager
...
time="2025-01-14T00:00:01Z" level=info msg="Node instance-manager-d80e13f520e7b952f4b7593fc1883e2a is marked unschedulable but removing harvester-node-1 PDB is blocked: some volumes are still attached InstanceEngines count 1 pvc-9ae0e9a5-a630-4f0c-98cc-b14893c74f9e-e-0" func="controller.(*InstanceManagerController).syncInstanceManagerPDB" file="instance_manager_controller.go:823" controller=longhorn-instance-manager node=harvester-node-1
----
+
O pod `instance-manager` não pode ser drenado devido ao motor `pvc-9ae0e9a5-a630-4f0c-98cc-b14893c74f9e-e-0`.

. Verifique se o motor ainda está em execução no nó preso.
+
Exemplo:
+
[,shell]
----
$ kubectl -n longhorn-system get engines.longhorn.io pvc-9ae0e9a5-a630-4f0c-98cc-b14893c74f9e-e-0 -o jsonpath='{"Current state: "}{.status.currentState}{"\nNode ID: "}{.spec.nodeID}{"\n"}'
Current state: stopped
Node ID:
----
+
O problema provavelmente existe se a saída mostrar que o motor não está em execução ou não foi encontrado.

. Verifique se todos os volumes estão saudáveis.
+
[,shell]
----
kubectl get volumes -n longhorn-system -o yaml | yq '.items[] | select(.status.state == "attached")| .status.robustness'
----
+
Todos os volumes devem estar marcados como `healthy`. Se este não for o caso, reporte o problema.

. Remova o PodDisruptionBudget (PDB) do pod `instance-manager`.
+
Exemplo:
+
[,shell]
----
kubectl delete pdb instance-manager-d80e13f520e7b952f4b7593fc1883e2a -n longhorn-system
----

== Falha na migração ao vivo no estado "Pré-drenado"

A migração ao vivo de máquinas virtuais pode falhar quando o nó em atualização estiver isolado durante o estado de pré-drenagem. Uma causa comum é a falta de nós-alvo compatíveis devido a regras de xref:virtual-machines/create-vm.adoc#_virtual_machine_scheduling[anti-affinity rigorosas].

Quando isso acontece, {harvester-product-name} desliga automaticamente essas máquinas virtuais para desbloquear a atualização e evitar que o processo reinicie de forma insegura.

== Snapshots e backups {longhorn-product-name} recorrentes não são suportados

https://longhorn.io/docs/1.10.0/snapshots-and-backups/scheduling-backups-and-snapshots[Snapshots e backups {longhorn-product-name} recorrentes] não estão integrados ao {harvester-product-name}. Se você decidir usar esse recurso, deve desativar todos os *trabalhos de instantâneo e backup recorrentes em {longhorn-product-name}* antes de iniciar a atualização.

Para mais informações sobre a incompatibilidade, veja xref:virtual-machines/backup-restore.adoc#_scheduled_virtual_machine_backups_and_snapshots[Backups e instantâneos de máquinas virtuais agendados].
