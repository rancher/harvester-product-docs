= Driver CSI do Harvester
:revdate: 2026-01-02
:page-revdate: {revdate}

O driver de Interface de Armazenamento de Contêineres (CSI) do Harvester fornece uma interface CSI padrão usada por clusters Kubernetes convidados. Ele se conecta ao cluster host e conecta volumes do host às máquinas virtuais para fornecer desempenho nativo de armazenamento.

[CAUTION]
====
Um https://github.com/harvester/harvester/issues/6849[problema conhecido] na v0.1.20 do Driver CSI do Harvester faz com que volumes fiquem presos quando o cluster host está executando uma {harvester-product-name} versão que foi lançada antes da v1.4.0.

Esse problema foi corrigido na v0.1.21. Se o seu sistema estiver afetado, você pode seguir a https://github.com/harvester/harvester/issues/6849#issuecomment-2462545795[solução alternativa] sugerida.

== Implantação

=== Pré-requisitos

* O cluster Kubernetes é construído sobre {harvester-product-name} máquinas virtuais.
* As {harvester-product-name} máquinas virtuais que funcionam como nós Kubernetes convidados estão no mesmo namespace.

[NOTE]
====
Atualmente, o Driver CSI do Harvester suporta apenas volumes de leitura e gravação (RWO) de nó único. Verifique https://github.com/harvester/harvester/issues/1992[o problema #1992] para informações sobre o possível suporte a volumes de somente leitura (ROX) e de leitura e gravação (RWX) em múltiplos nós.
====

=== Implantação com o driver de nó Harvester RKE2

Ao criar um cluster Kubernetes usando o driver de nó Rancher RKE2, o driver CSI do Harvester será implantado automaticamente quando o provedor de nuvem Harvester for selecionado.

image::rancher/rke2-cloud-provider.png[select-harvester-cloud-provider]

== Personalize a StorageClass padrão

Você pode usar o parâmetro `host-storage-class` para personalizar a StorageClass padrão.

. Crie uma StorageClass para o cluster Harvester host.

== Passthrough Custom StorageClass

A partir do driver CSI do Harvester v0.1.15, é possível criar um PersistentVolumeClaim (PVC) usando uma StorageClass do Harvester diferente no cluster Kubernetes convidado.

[NOTE]
====
Um driver CSI do Harvester compatível está embutido em cada versão suportada do RKE2.
====

=== Pré-requisitos

Adicione os seguintes pré-requisitos ao seu cluster Harvester para garantir que o driver CSI do Harvester exiba mensagens de erro corretamente. Configurações RBAC adequadas são essenciais para a visibilidade das mensagens de erro, especialmente ao criar um PVC com uma StorageClass inexistente, conforme mostrado na imagem abaixo:

image::rancher/error_event_csi_driver.png[]

Siga estas etapas para configurar *RBAC* para visibilidade das mensagens de erro:

. Crie um novo `clusterrole` chamado `harvesterhci.io:csi-driver` usando o seguinte manifesto.
+
----
 apiVersion: rbac.authorization.k8s.io/v1
 kind: ClusterRole
 metadata:
   labels:
     app.kubernetes.io/component: apiserver
     app.kubernetes.io/name: harvester
     app.kubernetes.io/part-of: harvester
   name: harvesterhci.io:csi-driver
 rules:
 - apiGroups:
   - storage.k8s.io
   resources:
   - storageclasses
   verbs:
   - get
   - list
   - watch
----

. Crie um novo `clusterrolebinding` associado ao `clusterrole` acima com o `serviceaccount` relevante usando o seguinte manifesto.
+
----
 apiVersion: rbac.authorization.k8s.io/v1
 kind: ClusterRoleBinding
 metadata:
   name: <namespace>-<serviceaccount name>
 roleRef:
   apiGroup: rbac.authorization.k8s.io
   kind: ClusterRole
   name: harvesterhci.io:csi-driver
 subjects:
 - kind: ServiceAccount
   name: <serviceaccount name>
   namespace: <namespace>
----
+
Certifique-se de que o `serviceaccount name` e o `namespace` correspondam às configurações do seu provedor de nuvem. Realize os seguintes passos para recuperar esses detalhes.

 .. Encontre o `rolebinding` associado ao seu provedor de nuvem:
+
----
 $ kubectl get rolebinding -A |grep harvesterhci.io:cloudprovider
 default                                 default-rke2-guest-01                                ClusterRole/harvesterhci.io:cloudprovider             7d1h
----

 .. Extraia as informações do `subjects` deste `rolebinding`:
+
----
 $ kubectl get rolebinding default-rke2-guest-01 -n default -o yaml |yq -e '.subjects'
----

 .. Identifique as informações do `ServiceAccount`:
+
----
 - kind: ServiceAccount
   name: rke2-guest-01
   namespace: default
----

=== Implantação

Agora você pode criar um novo StorageClass que pretende usar em seu cluster Kubernetes convidado.

. Para administradores, você pode criar um xref:../../storage/storageclass.adoc[StorageClass] desejado (por exemplo, nomeado *replica-2*) em seu cluster Harvester bare-metal.
+
image::rancher/sc-replica-2.png[]

. Em seguida, no cluster Kubernetes convidado, crie um novo StorageClass associado ao StorageClass nomeado *replica-2* do Cluster Harvester:
+
image::rancher/downstream-cluster-sc-creation.png[]
+

[NOTE]
====
 ** Ao escolher um *Provisioner*, selecione *Harvester (CSI)*. O parâmetro *Host StorageClass* deve corresponder ao nome do StorageClass criado no Cluster Harvester.
 ** Para proprietários de Kubernetes convidados, você pode solicitar que o administrador do cluster Harvester crie um novo StorageClass.
 ** Se você deixar o campo `Host StorageClass` vazio, o StorageClass padrão do cluster Harvester será utilizado.
====

. Agora você pode criar um PVC baseado neste novo *StorageClass*, que utiliza o *Host StorageClass* para provisionar volumes no cluster Harvester bare-metal.

== Suporte a Volume RWX

[CAUTION]
====
Volumes RWX atualmente funcionam apenas com uma rede de armazenamento dedicada. https://github.com/harvester/harvester/issues/7218[Issue #7218] rastreia a melhoria que permitirá que volumes RWX usem várias VLANs em clusters convidados.
====

=== Pré-requisitos

* Harvester v1.4 ou posterior está instalado no cluster host.

* Uma xref:../../networking/storage-network.adoc[rede de armazenamento] está configurada no cluster Harvester.
+
Use *exclude* para reservar um intervalo de endereços IP para as máquinas virtuais do cluster convidado. 
+
image::rancher/configure-storage-network-01.png[]

* A configuração *Rede de Armazenamento para Volume RWX* na interface Longhorn incorporada está habilitada. 
+
Vá para *Geral* e, em seguida, selecione *Rede de Armazenamento para Volume RWX Habilitado*.
+
image::rancher/enable-rwx-storage-network-01.png[]

* Você criou um StorageClass RWX no Harvester cluster do host.
+
Na *Classe de Armazenamento: Na tela de Criação*, clique em *Editar como YAML* e especifique o seguinte:
+
[,yaml]
----
kind: StorageClass
apiVersion: storage.k8s.io/v1
metadata:
  name: longhorn-rwx
provisioner: driver.longhorn.io
allowVolumeExpansion: true
reclaimPolicy: Delete
volumeBindingMode: Immediate
parameters:
  numberOfReplicas: "3"
  staleReplicaTimeout: "2880"
  fromBackup: ""
  fsType: "ext4"
  nfsOptions: "vers=4.2,noresvport,softerr,timeo=600,retrans=5"
----
+
image::rancher/create-rwx-sc-host-cluster-01.png[]
+
image::rancher/create-rwx-sc-host-cluster-02.png[]
+
image::rancher/create-rwx-sc-host-cluster-03.png[]

* As configurações de controle de acesso com base em função (RBAC) estão atualizadas.
+
https://kubernetes.io/docs/reference/access-authn-authz/rbac/[Autorização RBAC] usa um grupo de API Kubernetes específico para tomar decisões de autorização sobre o acesso a recursos computacionais ou de rede.
+
O driver CSI do Harvester requer as novas configurações de RBAC para suportar volumes RWX. Para verificar as configurações de RBAC, execute o comando `kubectl get clusterrole harvesterhci.io:csi-driver -o yaml`.
+
[,yaml]
----
# kubectl get clusterrole harvesterhci.io:csi-driver -o yaml
apiVersion: rbac.authorization.k8s.io/v1
kind: ClusterRole
metadata:
...
  name: harvesterhci.io:csi-driver
...
rules:
- apiGroups:
  - storage.k8s.io
  resources:
  - storageclasses
  verbs:
  - get
  - list
  - watch
- apiGroups:
  - harvesterhci.io
  resources:
  - networkfilesystems
  - networkfilesystems/status
  verbs:
  - '*'
- apiGroups:
  - longhorn.io
  resources:
  - volumes
  - volumes/status
  verbs:
  - get
  - list
----

* Os pods do networkfs-manager estão em execução.
+
Para verificar o status dos pods do networkfs-manager, execute o comando `kubectl get pods -n harvester-system | grep networkfs-manager`.
+
Exemplo:
+
[,yaml]
----
# kubectl get pods -n harvester-system | grep networkfs-manager
harvester-networkfs-manager-2pxhm                       1/1     Running     4 (34m ago)    3h41m
harvester-networkfs-manager-8tst2                       1/1     Running     4 (37m ago)    3h41m
harvester-networkfs-manager-xvkgp                       1/1     Running     4 (37m ago)    3h41m
----

* A versão do driver CSI do Harvester é v0.1.20 ou posterior.
+
image::rancher/harvester-csi-driver-version.png[]

* A máquina virtual possui duas interfaces de rede: uma interface de rede padrão para comunicações intra-cluster e para permitir acesso a partir da rede de infraestrutura (externa ao Harvester cluster); e uma interface de rede que pode se conectar à rede de armazenamento.
+
O NAD *default/vlan101* é usado para a rede de armazenamento.
+
image::rancher/create-guest-cluster-with-two-nics.png[]

* O cliente NFS está instalado em cada nó no cluster convidado.
+
Execute qualquer um dos seguintes comandos para instalar o cliente NFS.
+
** Debian e Ubuntu: `apt-get install -y nfs-common`
** CentOS e RHEL: `yum install -y nfs-utils`
** SUSE e OpenSUSE: `zypper install -y nfs-client`

* Um IP é atribuído manualmente à interface de rede de armazenamento.
+
Você pode atribuir qualquer um dos IPs reservados usando os seguintes comandos:
+
----
$ ip link set <storage network nic> up
$ ip a add <reserved IP> dev <storage network nic>
----
+
[IMPORTANT]
====
Um IP que é atribuído usando os comandos dados não persiste após uma reinicialização. Para tornar o IP persistente, você deve adicioná-lo ao arquivo de configuração de rede do seu sistema operacional convidado.
====

== Redimensionamento de volume online

Se o provedor de armazenamento subjacente suportar xref:storage/volumes/edit-volume.adoc#_online_volume_expansion[expansão de volume online], você pode expandir um volume ReadWriteOnce (RWO) no cluster convidado mesmo enquanto ele estiver anexado a uma carga de trabalho em execução.

=== Pré-requisitos

* {harvester-product-name} v1.6 ou posterior
* Harvester CSI Driver v0.1.24 ou posterior

== Instantâneos de Volume

A partir de *v0.1.25*, o Harvester CSI Driver suporta https://kubernetes.io/docs/concepts/storage/volume-snapshots/[instantâneos de volume], fornecendo capacidades de backup e restauração em um ponto no tempo para cargas de trabalho em execução em clusters Kubernetes convidados.

=== Pré-requisitos

* {harvester-product-name} v1.7 ou posterior
* Harvester CSI Driver v0.1.25 ou posterior
* O controlador de instantâneos CSI e os manifests necessários estão devidamente implantados no cluster convidado. Esses componentes são implantados por padrão no RKE2.

== Atualizar o CSI Driver

[NOTE]
====
O Harvester CSI Driver suporta instantâneos de volume a partir da versão *v0.1.25*. Usar este recurso pode exigir etapas adicionais, dependendo da sua distribuição Kubernetes.

* RKE2 (recomendado): O controlador de instantâneos CSI é implantado por padrão.
* Outras distribuições Kubernetes: Você deve instalar o controlador de instantâneos CSI no cluster convidado antes de atualizar o Harvester CSI Driver para *v0.1.25*. Para instruções de instalação, consulte a documentação da sua distribuição (exemplo: https://documentation.suse.com/cloudnative/storage/latest/en/snapshots-backups/csi-snapshots/enable-csi-snapshot-creation.html({longhorn-product-name}).)
====

=== Atualizar RKE2

Para atualizar o driver CSI, use a interface do Rancher para atualizar o RKE2. Certifique-se de que a nova versão do RKE2 suporte/venha com a versão atualizada do driver CSI.

. Vá para *☰* > *Gerenciamento de Cluster*.
. Encontre o cluster convidado que você deseja atualizar e selecione *⋮* > *Editar Config*.
. Selecione *Versão do Kubernetes*.
. Clique em *Salvar*.

=== Atualizar K3s

Você pode atualizar o K3s usando a interface do Rancher.

. Vá para *☰* > *Cluster K3s* > *Aplicativos* > *Aplicativos Instalados*.
. Encontre o gráfico do driver CSI e selecione *⋮* > *Editar/Atualizar*.
. Selecione *Versão*.
. Selecione *Próximo* > *Atualizar*.
