= 存储网络
:revdate: 2025-11-12
:page-revdate: {revdate}

{harvester-product-name}使用{longhorn-product-name}为虚拟机和pod提供块设备卷。如果您想将{longhorn-product-name}复制流量与`mgmt`（内置集群网络）或其他集群范围的工作负载隔离，可以使用专用存储网络以获得更好的网络带宽和性能。

有关更多信息，请参见 https://documentation.suse.com/cloudnative/storage/1.8/en/longhorn-system/networking/storage-network.html[存储网络]中的{longhorn-product-name}文档。

[NOTE]
====
避免直接配置{longhorn-product-name}设置，因为这可能导致意外或不希望的系统行为。
====

== 先决条件

在开始配置存储网络之前，请确保满足以下要求：

* 网络交换机已正确配置，并且为存储网络分配了专用VLAN ID。

* xref:networking/cluster-network.adoc[集群网络]和xref:networking/vm-network.adoc#_vlan_network[VLAN网络]已正确配置。确保这两个网络覆盖所有节点并且可访问。

* 存储网络的IP范围具有以下特征：
+
** 使用IPv4 CIDR格式
+
** 与Kubernetes集群网络不冲突或重叠
+
以下地址是保留的：`10.42.0.0/16`, `10.43.0.0/16`, `10.52.0.0/16`和`10.53.0.0/16`。
+
** 满足集群的要求
+
所需的IP地址数量使用以下公式计算：`Required number of IPs = (Number of nodes * 2) + (Number of disks * 2) + Number of images to be downloaded or uploaded`
+
示例：如果一个集群有五个节点，每个节点有两个磁盘，并且要同时上传十个镜像，则IP范围应大于或等于`/26`（计算：（5 x 2） + （5 x 2） + 10 = 30）。
+
** 排除{longhorn-product-name} pod和存储网络不得使用的IP地址，例如保留给xref:integrations/rancher/csi-driver.adoc#_rwx_volumes_support[RWX卷]、网关和其他组件的地址。

* Whereabouts CNI已正确安装。
+
您可以使用命令`ippools.whereabouts.cni.cncf.io`检查`kubectl get crd ippools.whereabouts.cni.cncf.io` CRD是否存在于集群中。
+
如果返回空字符串，请使用以下命令在 https://github.com/harvester/harvester/tree/v1.1.0/deploy/charts/harvester/dependency_charts/whereabouts/crds[此目录]中添加CRDs：
+
[,console]
----
kubectl apply -f https://raw.githubusercontent.com/harvester/harvester/v1.1.0/deploy/charts/harvester/dependency_charts/whereabouts/crds/whereabouts.cni.cncf.io_ippools.yaml
kubectl apply -f https://raw.githubusercontent.com/harvester/harvester/v1.1.0/deploy/charts/harvester/dependency_charts/whereabouts/crds/whereabouts.cni.cncf.io_overlappingrangeipreservations.yaml
----
+
[NOTE]
====
在某些 https://github.com/harvester/harvester/issues/3168[升级场景]中，CNI的安装不正确。
====

* 所有虚拟机已停止。
+
您可以使用命令`kubectl get -A vmi`检查虚拟机的状态，该命令应返回空字符串。
+
[CAUTION]
====
{harvester-product-name}通过{harvester-product-name} UI向已停止的虚拟机发送优雅的关闭信号。然而，工作负载会被中断，并且在您确认存储网络配置成功应用后，必须手动启动虚拟机才能恢复可用性。
====

* 所有附加到{longhorn-product-name}卷的pod已停止。

* 所有正在进行的镜像上传和下载要么已完成，要么已删除。

== {longhorn-product-name}复制流量路由

{longhorn-product-name}复制流量的路由取决于虚拟机VLAN流量和{longhorn-product-name}存储网络是否共享相同的物理接口或使用不同的接口。

* *相同的物理接口*：在以下示例中，`eth2`和`eth3`用于虚拟机VLAN流量，而{longhorn-product-name}存储网络也使用它们。红线表示{longhorn-product-name}通过`eth3`发送复制流量。
+

+
[NOTE]
====
您必须在集群网络和VLAN网络配置中包含`eth2`和`eth3`。
====

* *不同的物理接口*：在以下示例中，`eth2`和`eth3`用于虚拟机VLAN流量，而`eth4`和`eth5`用于{longhorn-product-name}存储网络。红线表示{longhorn-product-name}通过`eth4`发送复制流量。
+

+
[NOTE]

您必须在集群网络和VLAN网络配置中包含`eth4`和`eth5`。

== `storage-network`设置

xref:../installation-setup/config/settings.adoc#_storage_network[`storage-network`]设置允许您配置在需要隔离时用于隔离集群内存储流量的网络。

您可以使用用户界面或CLI启用和禁用存储网络。当设置启用时，您必须通过配置某些字段来构建Multus `NetworkAttachmentDefinition` CRD。

一旦应用了`storage-network`设置，{harvester-product-name}将执行以下操作：

* 停止与{longhorn-product-name}卷、Prometheus、Grafana、Alertmanager和VM导入控制器相关的所有pod。
* 创建一个新的`NetworkAttachmentDefinition`并更新{longhorn-product-name}存储网络设置。
* 重启所有`instance-manager`和`backing-image-manager`pod以应用新的配置。

=== 配置步骤

[tabs]
======
UI::
+
--
[TIP]
====
强烈建议使用{harvester-product-name}用户界面来配置`storage-network`设置。
====

==== 启用存储网络

. 前往*高级 -> 设置 -> 存储网络*。

. 选择*启用*。

. 配置*VLAN ID*、*集群网络*、*IP范围*和*排除*字段以构建Multus `NetworkAttachmentDefinition` CRD。

. 单击*保存*。

image::storagenetwork/storage-network-enabled.png[存储网络已启用]

==== 禁用存储网络

. 前往*高级 -> 设置 -> 存储网络*。

. 选择*禁用*。

. 单击*保存*。

一旦存储网络被禁用，{longhorn-product-name}将开始使用pod网络进行存储相关操作。

image::storagenetwork/storage-network-disabled.png[存储网络已禁用]
--

CLI::
+
--
您可以使用以下命令来配置xref:../installation-setup/config/settings.adoc#_storage_network[`storage-network`设置]。

[,bash]
----
kubectl edit settings.harvesterhci.io storage-network
----

在以下情况下，存储网络会自动启用：

* `value`字段包含有效的JSON字符串。
+
[,yaml]
----
apiVersion: harvesterhci.io/v1beta1
kind: Setting
metadata:
  name: storage-network
value: '{"vlan":100,"clusterNetwork":"storage","range":"192.168.0.0/24", "exclude":["192.168.0.100/32"]}'
----

* `value`字段为空。
+
[,yaml]
----
apiVersion: harvesterhci.io/v1beta1
kind: Setting
metadata:
  name: storage-network
value: ''
----

当您移除`value`字段时，存储网络将被禁用。

[,yaml]
----
apiVersion: harvesterhci.io/v1beta1
kind: Setting
metadata:
  name: storage-network
----

[CAUTION]
====
{harvester-product-name} 将 JSON 字符串中的额外无关字符视为不同的配置。
====

== 最佳实践

* 在为存储网络配置IP范围时，请确保分配的IP地址能够满足集群未来的需求。这很重要，因为{longhorn-product-name} pods（`instance-manager`和`backing-image-manager`）在向集群添加新节点或在存储网络配置后向节点添加更多磁盘时停止运行，并且当所需的IP数量超过分配的IP时。解决此问题涉及使用正确的IP范围重新配置存储网络。
+
{longhorn-product-name} pods 使用存储网络如下：
+
** `instance-manager` pods：实例管理器组件被 https://longhorn.io/docs/1.5.0/deploy/important-notes/#instance-managers-consolidated[在 {longhorn-product-name} v1.5.0 中合并]。每个节点需要一个 IP 地址。在升级期间，这些 pods 的旧版本和新版本同时存在，旧版本在升级完成后被删除。
+
** `backing-image-ds` pods：这些 pods 处理即时上传和下载的底层镜像数据源，并在镜像上传和下载完成后被去除。
+
** `backing-image-manager` pods：每个磁盘需要一个 IP 地址。在升级期间，这些 pods 的旧版本和新版本同时存在，旧版本在升级完成后被删除。

* 在非-`mgmt` 集群网络上配置存储网络，以确保 {longhorn-product-name} 复制流量与 Kubernetes 控制平面流量完全隔离。使用 `mgmt` 是可能的，但不推荐，因为它会对控制平面网络性能产生负面影响（资源和带宽争用）。仅在您的集群存在与 NIC 相关的限制，并且能够完全隔离流量时使用 `mgmt`。
