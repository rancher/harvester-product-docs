= {rancher-product-name} Integration
:revdate: 2025-09-18
:page-revdate: {revdate}

https://documentation.suse.com/cloudnative/rancher-manager[{rancher-product-name}] ist eine Open Source-Plattform für das Management von Multi-Clustern. {rancher-short-name} hat standardmäßig {harvester-product-name} integriert, um virtuelle Maschinen und Container zentral zu verwalten.

Sie können mehrere {harvester-product-name} Cluster mit der {rancher-short-name} Funktion von xref:./virtualization-management.adoc[Virtualisierungsmanagement] importieren und verwalten, die die {rancher-short-name} https://documentation.suse.com/cloudnative/rancher-manager/v2.11/en/rancher-admin/users/authn-and-authz/authn-and-authz.html[Authentifizierungs] Funktion und https://documentation.suse.com/cloudnative/rancher-manager/v2.11/en/rancher-admin/users/authn-and-authz/manage-role-based-access-control-rbac/manage-role-based-access-control-rbac.html[RBAC-Kontrolle] für die Unterstützung von xref:./virtualization-management.adoc#_multi_tenancy[Multi-Tenancy] nutzt.

Überprüfen Sie die xref:../../installation-setup/requirements.adoc#_network_requirements[Netzwerkanforderungen], bevor Sie irgendwelche Implementierungsschritte durchführen.

== Bereitstellen des {rancher-short-name} Servers

Um {rancher-short-name} mit {harvester-product-name} zu verwenden, müssen Sie {rancher-short-name} auf einem separaten Server installieren. Wenn Sie die Integrationsfunktionen ausprobieren möchten, können Sie eine virtuelle Maschine in {harvester-product-name} erstellen und den {rancher-short-name} Server installieren, indem Sie der https://documentation.suse.com/cloudnative/rancher-manager/v2.11/en/installation-and-upgrade/quick-start/deploy-rancher/helm-cli.html[Helm CLI Kurzanleitung] folgen.

Für Produktionsumgebungen können Sie https://documentation.suse.com/cloudnative/rancher-manager/v2.11/en/installation-and-upgrade/quick-start/deploy-rancher/deploy-rancher.html[{rancher-short-name} bereitstellen und ein Kubernetes-Cluster bei verschiedenen Cloud-Anbietern bereitstellen].

Wenn Sie {rancher-short-name} lokal oder mit einem Anbieter, der nicht in der Dokumentation aufgeführt ist, ausführen möchten, können Sie https://documentation.suse.com/cloudnative/rancher-manager/v2.11/en/installation-and-upgrade/other-installation-methods/http-proxy/install-rancher.html#_install_the_helm_cli[die Helm CLI installieren] und dann {rancher-short-name} mit Helm installieren.

[CAUTION]
====
*Installieren Sie {rancher-short-name} nicht mit Docker in der Produktion*. Andernfalls könnte Ihre Umgebung beschädigt werden, und Ihr Cluster könnte nicht wiederhergestellt werden. Die Installation von {rancher-short-name} in Docker sollte nur für schnelle Bewertungen und Testzwecke verwendet werden.
====

== Virtualisierungsmanagement

Mit der Virtualisierungsmanagement-Funktion von {rancher-short-name} können Sie Ihren {harvester-product-name} Cluster importieren und verwalten. Durch Klicken auf einen der importierten Cluster können Sie einfach auf eine Reihe von {harvester-product-name} Cluster-Ressourcen zugreifen und diese verwalten, einschließlich Hosts, virtuellen Maschinen, Images, Volumes und mehr. Darüber hinaus nutzt die Virtualisierungsmanagement-Funktion die bestehenden Fähigkeiten von {rancher-short-name}, wie Authentifizierung mit verschiedenen Auth-Anbietern und Unterstützung für Multi-Tenancy.

Für tiefere Einblicke konsultieren Sie bitte die xref:./virtualization-management.adoc[Seite zum Virtualisierungsmanagement].

image::rancher/import-harvester-cluster.png[Cluster importieren]

== Erstellen von Kubernetes-Clustern mit dem Harvester-Knotentreiber

Sie können ein Kubernetes-Cluster von {rancher-short-name} mit dem xref:../../integrations/rancher/node-driver/node-driver.adoc[Harvester-Knotentreiber] starten. Wenn {rancher-short-name} Kubernetes auf diesen Knoten bereitstellt, wählen Sie RKE2.

Ein Vorteil der Installation von Kubernetes auf Knotenpools, die vom Knotentreiber gehostet werden, besteht darin, dass {rancher-short-name} automatisch einen weiteren Knoten erstellen kann, um dem Cluster beizutreten, falls ein Knoten die Verbindung zum Cluster verliert, um sicherzustellen, dass die Anzahl des Knotenpools wie erwartet ist.

Der Harvester-Knotentreiber ist standardmäßig in {rancher-short-name} enthalten. Für weitere Informationen, siehe xref:../../integrations/rancher/node-driver/node-driver.adoc[Knotentreiber].

== Bare-Metal-Container-Workload-Unterstützung (Experimentell)

{harvester-product-name} ermöglicht es Ihnen, Container-Workloads direkt auf dem zugrunde liegenden {harvester-product-name} Cluster bereitzustellen und zu verwalten. Mit dieser Funktion können Sie die Leistung von virtuellen Maschinen nahtlos mit der Flexibilität der Containerisierung kombinieren, was eine vielseitigere und effizientere Infrastruktur ermöglicht.

image::rancher/harvester-container-dashboard.png[harvester-container-dashboard]

Dieser Leitfaden führt Sie durch die Aktivierung und Nutzung dieser experimentellen Funktion und hebt deren Möglichkeiten und bewährte Praktiken hervor.

Um dieses neue Funktionsflag zu aktivieren, befolgen Sie diese Schritte:

. Klicken Sie auf das Hamburger-Menü und wählen Sie die Registerkarte *Globale Einstellungen*.
. Klicken Sie auf *Funktionsflags* und suchen Sie das neue Funktionsflag `harvester-baremetal-container-workload`.
. Klicken Sie auf das Dropdown-Menü und wählen Sie *Aktivieren*, um diese Funktion zu aktivieren.
. Wenn sich der Status der Funktion auf *Aktiv* ändert, ist die Funktion erfolgreich aktiviert.

image::rancher/harvester-baremetal-container-workload-feature.png[harvester-baremetal-container-workload-feature]

=== Die wichtigsten Funktionen

* *Vereinheitlichte Dashboard-Ansicht*: Sobald Sie die Funktion aktiviert haben, können Sie die Dashboard-Ansicht des {harvester-product-name} Clusters erkunden, genau wie bei anderen Standard-Kubernetes-Clustern. Diese einheitliche Erfahrung vereinfacht die Verwaltung und Überwachung sowohl Ihrer virtuellen Maschinen als auch Ihrer Container-Workloads über eine einzige, benutzerfreundliche Oberfläche.

* *Benutzerdefinierte Workloads bereitstellen*: Dieses Feature ermöglicht es Ihnen, benutzerdefinierte Container-Workloads direkt auf dem Bare-Metal {harvester-product-name} Cluster bereitzustellen. Obwohl diese Funktionalität experimentell ist, eröffnet sie spannende Möglichkeiten zur Optimierung Ihrer Infrastruktur. Wir empfehlen jedoch, Container- und virtuelle Maschinen-Workloads in separaten Namespaces bereitzustellen, um Klarheit und Trennung zu gewährleisten.

[NOTE]
====
* Kritische Systemkomponenten wie Überwachung, Protokollierung, {rancher-short-name}, KubeVirt und Longhorn werden alle vom {harvester-product-name} Cluster selbst verwaltet. Sie können diese Komponenten nicht upgraden oder ändern. Daher sollten Sie vorsichtig sein und Änderungen an diesen kritischen Systemkomponenten vermeiden.
* Es ist wichtig, keine Workloads in den System-Namespaces `cattle-system`, `harvester-system` oder `longhorn-system` bereitzustellen. Die Aufrechterhaltung Ihrer Workloads in separaten Namespaces ist entscheidend, um Klarheit zu bewahren und die Integrität der Systemkomponenten zu sichern.
* Es wird empfohlen, Container- und virtuelle Maschinen-Workloads in separaten Namespaces bereitzustellen.
====

== {fleet-product-name} Unterstützung (Experimentell)

Sie können https://documentation.suse.com/cloudnative/continuous-delivery/index.html[{fleet-product-name}] nutzen, um Container-Workloads zu verwalten und {harvester-product-name} mit einem GitOps-basierten Ansatz zu konfigurieren.

[IMPORTANT]
====
Die {rancher-short-name} Funktion `harvester-baremetal-container-workload` muss aktiviert sein.
====

. Gehen Sie in der {rancher-short-name} Benutzeroberfläche zu *☰ -> Continuous Delivery*.
+
image::rancher/continuous-delivery-overview.png[]

. (Optional) Gehen Sie auf der Registerkarte *Clusters*, um die Fleet-Cluster-Konfiguration zu bearbeiten und Labels hinzuzufügen, die verwendet werden können, um {harvester-product-name} Cluster zu gruppieren.
+
In diesem Beispiel wurde das Label `location=private-dc` hinzugefügt.
+
image::rancher/fleet-cluster-config.png[]
+
image::rancher/fleet-additional-labels.png[]

. (Optional) Gehen Sie auf der Registerkarte *Clustergruppen*, um eine Clustergruppe zu erstellen.
+
In diesem Beispiel wird die Clustergruppe `private-dc-clusters` mit einer Cluster-Selektorregel erstellt, die dem Label-Schlüssel/Wert-Paar von `location=private-dc` entspricht.
+
image::rancher/create-cluster-group.png[]

. Gehen Sie auf der Registerkarte *Git-Repos*, um ein Git-Repository mit dem Namen `harvester-config` zu erstellen, das auf das https://github.com/harvester/harvester-fleet-examples[harvester-fleet-examples Repository] verweist, wobei der Branch als `main` definiert ist. Sie müssen die folgenden Pfade definieren:
+
* `keypair`
* `vmimage`
* `vmnetwork`
* `cloudinit`
+
image::rancher/gitrepo-definition.png[]

. Klicken Sie auf *Weiter* und definieren Sie dann die Ziele des Git-Repositories. Sie können alle Cluster, einen einzelnen Cluster oder eine Gruppe von Clustern auswählen.
+
In diesem Beispiel wird die Clustergruppe mit dem Namen `private-dc-clusters` verwendet.
+
image::rancher/gitrepo-targets.png[]

. Klicken Sie auf *Speichern*. Es kann einige Sekunden dauern, bis die Ressourcen auf die Zielcluster bereitgestellt werden.
+
image::rancher/gitrepo-synced.png[]
