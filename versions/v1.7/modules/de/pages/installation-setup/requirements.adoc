= Hardware- und Netzwerkanforderungen
:revdate: 2025-12-30
:page-revdate: {revdate}

Als HCI-Lösung auf Bare-Metal-Servern gibt es Mindestanforderungen an die Hardware und das Netzwerk für die Installation und den Betrieb von {harvester-product-name}.

Ein Cluster mit drei Knoten ist erforderlich, um die Funktionen mit mehreren Knoten vollständig zu nutzen. Der erste Knoten, der dem Cluster hinzugefügt wird, ist standardmäßig der Verwaltungsknoten. Wenn das Cluster drei oder mehr Knoten hat, werden die zwei Knoten, die nach dem ersten hinzugefügt werden, automatisch zu Verwaltungsknoten befördert, um ein Hochverfügbarkeitscluster (HA) zu bilden.

Die neuesten Versionen unterstützen die Bereitstellung von xref:installation-setup/single-node-clusters.adoc[Einzelknoten-Clustern]. Solche Cluster unterstützen keine hohe Verfügbarkeit, mehrere Replikate und Live-Migration.

[IMPORTANT]
====
* Gemischte Architekturen werden nicht unterstützt. Stellen Sie separate Cluster bereit, um unerwartetes Systemverhalten zu vermeiden.
* Für die besten Ergebnisse verwenden Sie https://www.suse.com/yesCertified/home#search?platforms=1361&platforms=1360&platforms=1355[YES-zertifizierte Hardware] für SUSE Linux Micro 5.5/6.0/6.1. {harvester-product-name} basiert auf SUSE Linux Enterprise-Technologie und YES-zertifizierte Hardware hat eine zusätzliche Validierung der Treiber- und Systemplatinenkompatibilität. Laptops und verschachtelte Virtualisierung werden nicht unterstützt.
* Verschachtelte Virtualisierung wird auf virtuellen Maschinen, die auf {harvester-product-name} laufen, nicht unterstützt.
* Jeder Knoten muss eine eindeutige `product_uuid` (abgerufen von `/sys/class/dmi/id/product_uuid`) haben, um Fehler während der Live-Migration von VMs und anderen Vorgängen zu vermeiden. Für weitere Informationen siehe https://github.com/harvester/harvester/issues/4025[Issue #4025].
* {harvester-product-name} hat ein xref:networking/cluster-network.adoc#_built_in_cluster_network[integriertes Management-Cluster-Netzwerk] (`mgmt`). Um hohe Verfügbarkeit und die beste Leistung in Produktionsumgebungen zu erreichen, verwenden Sie mindestens zwei NICs in jedem Knoten, um eine gebündelte NIC für das Management-Netzwerk einzurichten (siehe Schritt 6 in xref:installation-setup/methods/iso-install.adoc#_installation_steps[ISO-Installation]). Sie können auch xref:networking/cluster-network.adoc#_custom_cluster_network[benutzerdefinierte Cluster-Netzwerke] für VM-Workloads erstellen. Jedes benutzerdefinierte Cluster-Netzwerk benötigt mindestens zwei zusätzliche NICs, um eine gebündelte NIC in jedem beteiligten Knoten des Clusters einzurichten. Der xref:hosts/witness-node.adoc[witness node] benötigt keine zusätzlichen NICs. Für weitere Informationen siehe xref:networking/cluster-network.adoc#_concepts[Cluster-Netzwerk].
* Während der Tests können Sie nur eine NIC für das xref:networking/cluster-network.adoc#_built_in_cluster_network[integrierte Management-Cluster-Netzwerk] (`mgmt`) verwenden und zum Testen des xref:networking/vm-network.adoc#_create_a_vm_network[VM-Netzwerks], das ebenfalls von `mgmt` getragen wird. Hohe Verfügbarkeit und optimale Leistung sind nicht garantiert.
* Wenn die Festplatte nur die minimal erforderliche Kapazität erfüllt, können während des Upgrades Probleme im Zusammenhang mit dem xref:upgrades/upgrades.adoc#_free_system_partition_space_requirement[freien Speicherplatzbedarf der Systempartition] auftreten.
* Die Unterstützung für das Booten mit Legacy-BIOS läuft in v1.7.0 aus und wird in einer späteren Version entfernt. Bestehende {harvester-product-name} Cluster, die diesen Bootmodus verwenden, werden weiterhin funktionieren, aber ein Upgrade auf spätere Versionen kann eine Neuinstallation im UEFI-Modus erfordern. Um Probleme und Unterbrechungen zu vermeiden, verwenden Sie UEFI bei neuen Installationen.

=== CPU-Spezifikationen

xref:virtual-machines/live-migration.adoc[Live-Migration] funktioniert nur korrekt, wenn die CPUs aller physischen Server im Cluster die gleichen Spezifikationen haben. Dieses Erfordernis gilt für alle Vorgänge, die auf der Live-Migrationsfunktionalität basieren, wie z.B. die automatische VM-Migration, wenn xref:hosts/hosts.adoc#_node_maintenance[Wartungsmodus] aktiviert ist.

Neuere CPUs (auch solche vom gleichen Anbieter, Generation und Familie) können unterschiedliche Fähigkeiten haben, die möglicherweise den VM-Betriebssystemen zur Verfügung stehen. Um die Stabilität der VM zu gewährleisten, überprüft die Live-Migration, ob die CPU-Fähigkeiten konsistent sind, und blockiert Migrationsversuche, wenn die Quelle und das Ziel inkompatibel sind.

Beim Erstellen von Clustern, Hinzufügen weiterer Hosts zu einem Cluster und Ersetzen von Hosts sollten Sie immer CPUs mit den gleichen Spezifikationen verwenden, um betriebliche Einschränkungen zu vermeiden.

=== Portanforderungen für die Integration mit {rancher-product-name}

Wenn Sie xref:integrations/rancher/rancher-integration.adoc[mit {rancher-product-name} integrieren] möchten, müssen Sie sicherstellen, dass alle {harvester-product-name} Knoten eine Verbindung zum TCP-Port *443* des {rancher-product-name} Load Balancers herstellen können.

Beim Bereitstellen von VMs mit Kubernetes-Clustern von {rancher-product-name} nach {harvester-product-name} müssen Sie in der Lage sein, eine Verbindung zum TCP-Port *443* des {rancher-product-name} Load Balancers herzustellen. Andernfalls kann der Cluster nicht von {rancher-product-name} verwaltet werden. Für weitere Informationen siehe https://documentation.suse.com/cloudnative/rancher-manager/v2.11/en/about-rancher/architecture/communicating-with-downstream-clusters.html[Rancher-Architektur].

== Zeitanforderungen

Ein zuverlässiger Network Time Protocol (NTP)-Server ist entscheidend für die Aufrechterhaltung der korrekten Systemzeit über alle Knoten in einem Kubernetes-Cluster, insbesondere beim Betrieb von {harvester-product-name}. Kubernetes ist auf etcd angewiesen, einen verteilten Schlüssel-Wert-Speicher, der eine präzise Zeitsynchronisation erfordert, um die Datenkonsistenz sicherzustellen und Probleme mit der Führungswahl, der Protokollreplikation und der Clusterstabilität zu vermeiden.

Die Gewährleistung einer genauen und konsistenten Zeit im gesamten Cluster ist entscheidend für Zuverlässigkeit, Sicherheit und die allgemeine Systemintegrität.
