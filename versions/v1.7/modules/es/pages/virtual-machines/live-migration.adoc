= Migraciﾃｳn en vivo
:revdate: 2025-12-30
:page-revdate: {revdate}

La migraciﾃｳn en vivo significa mover una mﾃ｡quina virtual a un host diferente sin tiempo de inactividad. Un par de procesos y tareas exhaustivas se realizan en segundo plano para llevar a cabo la migraciﾃｳn en vivo.

== Requisitos previos

La migraciﾃｳn en vivo puede ocurrir cuando se cumplen los siguientes requisitos:

* Ademﾃ｡s del nodo actual, el clﾃｺster tiene al menos un nodo programable que coincide con todas las reglas de programaciﾃｳn de la mﾃ｡quina virtual.

* El nodo de destino de la migraciﾃｳn tiene suficientes recursos disponibles para alojar la mﾃ｡quina virtual.

* La CPU, la memoria, xref:virtual-machines/create-vm.adoc#_volumes[volﾃｺmenes], dispositivos y otros recursos solicitados por la mﾃ｡quina virtual pueden ser copiados o reconstruidos en el nodo de destino de la migraciﾃｳn mientras la mﾃ｡quina virtual de origen sigue funcionando.

== Mﾃ｡quinas Virtuales No Migrables

Una mﾃ｡quina virtual se considera no migrable si tiene una o mﾃ｡s de las siguientes:

* Volumen con las siguientes propiedades:

** Tipo: `CD-ROM` o `Container Disk`
** Modo de acceso: `ReadWriteOnce`
** Recuento de rﾃｩplicas de StorageClass: `1` (Esto no se detecta en todos los casos.)

* Dispositivos de host en modo passthrough como `PCI` y `vGPU`

* xref:virtual-machines/create-vm.adoc#_node_scheduling[Selector de nodo] que vincula la mﾃ｡quina virtual a un nodo especﾃｭfico

* xref:virtual-machines/create-vm.adoc#_node_scheduling[Reglas de programaciﾃｳn] que coinciden solo con un nodo.

Los siguientes son ejemplos de condiciones de regla que se verifican en tiempo de ejecuciﾃｳn:

* La mﾃ｡quina virtual estﾃ｡ conectada a una red de clﾃｺster que cubre solo un nodo.

* El pinning de CPU estﾃ｡ habilitado en la mﾃ｡quina virtual, y el Administrador de CPU solo estﾃ｡ habilitado en un nodo.

* La mﾃ｡quina virtual tiene reglas de anti-afinidad estrictas que impiden que se co-localice con ciertas otras mﾃ｡quinas virtuales.

Para mﾃ｡s informaciﾃｳn, consulta xref:virtual-machines/create-vm.adoc#_automatically_applied_affinity_rules[Reglas de Afinidad Aplicadas Automﾃ｡ticamente].

[NOTE]
====
Para realizar la migraciﾃｳn en vivo de la mﾃ｡quina virtual, primero debes eliminar los dispositivos no migrables y aﾃｱadir nodos programables.
====

== Mﾃ｡quinas Virtuales Migrables en Vivo

Las mﾃ｡quinas virtuales que no tienen las propiedades de mﾃ｡quinas virtuales no migrables pueden ser migradas en vivo.

== Cﾃｳmo Funciona la Migraciﾃｳn

=== `VirtualMachineInstanceMigration` Objeto

Cuando se activa una acciﾃｳn de migraciﾃｳn de mﾃ｡quina virtual, se crea un `VirtualMachineInstanceMigration` objeto para rastrear el estado y el progreso de la operaciﾃｳn. El controlador {harvester-product-name} correlaciona el `VirtualMachineInstanceMigration` objeto con el `VirtualMachineInstance` objeto asegurando que la identidad del objeto de instancia se refleje en el objeto de migraciﾃｳn.

En el siguiente ejemplo, la mﾃ｡quina virtual llamada `demo` tiene un objeto de migraciﾃｳn asociado. El UID de este objeto se aﾃｱade a la propiedad `.status.migrationState.migrationUID` del objeto de instancia durante la migraciﾃｳn.

[bash, shell]
----
$ kubectl get vmi demo -ojsonpath={.status.migrationState.migrationUID}

1d6d7273-275d-48e0-bb76-62e240b42aaf
----

El nombre del objeto de instancia se aﾃｱade a la propiedad `.spec.vmiName` del objeto de migraciﾃｳn.

[bash, shell]
----
$ kubectl get vmim demo-6crrk -ojsonpath={.spec.vmiName}

demostraciﾃｳn
----

El formato del nombre del objeto `VirtualMachineInstanceMigration` varﾃｭa dependiendo de si la migraciﾃｳn es activada manualmente o automﾃ｡ticamente.

Cuando se activa una migraciﾃｳn desde el **Migrar** xref:#_starting_a_migration[elemento del menﾃｺ], el nombre del objeto `VirtualMachineInstanceMigration` se precede con el nombre de la mﾃ｡quina virtual y una cadena aleatoria (por ejemplo, `vm1-a3d1f`).

Cuando se activa una migraciﾃｳn xref:#_automatically_triggered_batch_migration[automﾃ｡ticamente], al nombre del objeto `VirtualMachineInstanceMigration` se le antepone `kubevirt-evacuation-` y una cadena aleatoria (por ejemplo, `kubevirt-evacuation-9c485`).

[NOTE]
====
La interfaz de usuario {harvester-product-name} no especifica la fuente de la migraciﾃｳn. Debe comprobar el nombre del objeto `VirtualMachineInstanceMigration` para recuperar esta informaciﾃｳn.
====

=== Coincidencia del Modelo de CPU

Cada nodo tiene mﾃｺltiples modelos de CPU que estﾃ｡n etiquetados con diferentes claves.

* Modelo de CPU principal: `host-model-cpu.node.kubevirt.io/{cpu-model}`
* Modelos de CPU compatibles: `cpu-model.node.kubevirt.io/{cpu-model}`
* Modelos de CPU compatibles para migraciﾃｳn: `cpu-model-migration.node.kubevirt.io/{cpu-model}`

Durante la migraciﾃｳn en vivo, el controlador verifica el valor de `spec.domain.cpu.model` en el CR de VirtualMachineInstance (VMI), que se deriva de `spec.template.spec.domain.cpu.model` en el CR de VirtualMachine (VM). Si el valor de `spec.template.spec.domain.cpu.model` no estﾃ｡ establecido, el controlador utiliza el valor predeterminado `host-model`.

Cuando se utiliza `host-model`, el proceso obtiene el valor del modelo de CPU principal y llena `spec.NodeSelectors` del pod reciﾃｩn creado con la etiqueta `cpu-model-migration.node.kubevirt.io/{cpu-model}`.

Alternativamente, puedes personalizar el modelo de CPU en `spec.domain.cpu.model`. Por ejemplo, si el modelo de CPU es `XYZ`, el proceso llena `spec.NodeSelectors` del pod reciﾃｩn creado con la etiqueta `cpu-model.node.kubevirt.io/XYZ`.

Sin embargo, `host-model` solo permite la migraciﾃｳn de la VM a un nodo con el mismo modelo de CPU. Para obtener mﾃ｡s informaciﾃｳn, consulte xref:#_limitations[Limitaciones].

== Migraciﾃｳn por lotes activada automﾃ｡ticamente

xref:virtual-machines/live-migration.adoc#_live_migratable_virtual_machines[Actualizar versiﾃｳn] y xref:hosts/hosts.adoc#_node_maintenance[mantenimiento de nodos] se benefician de la migraciﾃｳn en vivo. El proceso subyacente, que se llama *migraciﾃｳn por lotes*, es ligeramente diferente del que se describe en xref:#_starting_a_migration[Iniciar una migraciﾃｳn]. Este proceso implica los siguientes pasos:

. El controlador observa una marca dedicada en el objeto nodo.

. El controlador crea un objeto `VirtualMachineInstanceMigration` para cada xref:#_live_migratable_virtual_machines[mﾃ｡quina virtual migrable en vivo] en el nodo actual.

. Las migraciones se ponen en cola, se programan internamente y se procesan en lotes. La interfaz de usuario {harvester-product-name} muestra los estados **Migraciﾃｳn pendiente** y **Migrando** para indicar el progreso.

. El controlador supervisa el procesamiento y espera hasta que todos se completen o hayan expirado.

== Tiempos de espera de migraciﾃｳn

=== Tiempo de espera de finalizaciﾃｳn

El proceso de migraciﾃｳn en vivo copia las pﾃ｡ginas de memoria de la mﾃ｡quina virtual y los bloques de disco al destino. En ciertos casos, la mﾃ｡quina virtual puede escribir en diferentes pﾃ｡ginas de memoria o bloques de disco a una tasa mﾃ｡s alta de la que se pueden copiar. Como resultado, se impide que el proceso de migraciﾃｳn se complete en un tiempo razonable.

La migraciﾃｳn en vivo se abortarﾃ｡ si supera el tiempo de finalizaciﾃｳn de 800s por GiB de datos. Por ejemplo, una mﾃ｡quina virtual con 8 GiB de memoria supera el tiempo de espera despuﾃｩs de 6400 segundos.

=== Tiempo de espera de progreso

La migraciﾃｳn en vivo tambiﾃｩn se aborta cuando la copia de la memoria no avanza en 150s.

== limitaciones

=== Modelos de CPU

`host-model` solo permite la migraciﾃｳn de la mﾃ｡quina virtual a un nodo con el mismo modelo de CPU. Sin embargo, especificar un modelo de CPU no siempre es necesario. Cuando no se especifica un modelo de CPU, debes apagar la mﾃ｡quina virtual, asignar un modelo de CPU que sea compatible con todos los nodos y luego reiniciar la mﾃ｡quina virtual.

Ejemplo:

* Nodo A: `host-model-cpu.node.kubevirt.io/XYZ` `cpu-model-migration.node.kubevirt.io/XYZ` `cpu-model.node.kubevirt.io/123`
* Nodo B: `host-model-cpu.node.kubevirt.io/ABC` `cpu-model-migration.node.kubevirt.io/ABC` `cpu-model.node.kubevirt.io/123`

No es posible migrar una mﾃ｡quina virtual con `host-model` porque los valores de `host-model-cpu.node.kubevirt.io` no son idﾃｩnticos. Sin embargo, ambos nodos son compatibles con el modelo de CPU `123`, por lo que puedes migrar cualquier mﾃ｡quina virtual con el modelo de CPU `123` utilizando cualquiera de los siguientes mﾃｩtodos:

* Nivel de clﾃｺster: Ejecuta `kubectl edit kubevirts.kubevirt.io -n harvester-system` y aﾃｱade `spec.configuration.cpuModel: "123"`. Este cambio tambiﾃｩn afecta a las mﾃ｡quinas virtuales reciﾃｩn creadas.
* Mﾃ｡quinas virtuales individuales: Modifica la configuraciﾃｳn de la mﾃ｡quina virtual para incluir `spec.template.spec.domain.cpu.model: "123"`.

Ambos mﾃｩtodos requieren que reinicies las VMs. Si estﾃ｡s seguro de que todos los nodos en el clﾃｺster son compatibles con un modelo de CPU especﾃｭfico, puedes definir esto a nivel de clﾃｺster antes de crear cualquier VM. Al hacerlo, eliminas la necesidad de reiniciar las VMs (para asignar el modelo de CPU) durante la migraciﾃｳn en vivo.

=== Interrupciones de red

La migraciﾃｳn en vivo es muy sensible a las interrupciones de red. Cualquier interrupciﾃｳn en la conexiﾃｳn de red entre los nodos de origen y destino durante la migraciﾃｳn puede tener una variedad de resultados.

==== `mgmt` interrupciones de red

La migraciﾃｳn en vivo a travﾃｩs de `mgmt` (la red de clﾃｺster integrada) depende de la disponibilidad de las interfaces de gestiﾃｳn en los nodos de origen y destino. Las `mgmt` interrupciones de red se consideran crﾃｭticas porque no solo interrumpen el proceso de migraciﾃｳn, sino que tambiﾃｩn afectan la gestiﾃｳn general de los nodos.

==== Interrupciones de red de migraciﾃｳn de VM

Una xref:networking/vm-network.adoc[red de migraciﾃｳn de VM] aﾃｭsla el trﾃ｡fico de migraciﾃｳn de otras actividades de red. Si bien esta configuraciﾃｳn mejora el rendimiento y la fiabilidad de la migraciﾃｳn, especialmente en entornos con alto trﾃ｡fico de red, tambiﾃｩn hace que el proceso de migraciﾃｳn dependa de la disponibilidad de esa red especﾃｭfica.

Una interrupciﾃｳn en la red de migraciﾃｳn de VM puede afectar el proceso de migraciﾃｳn de las siguientes maneras:

* Interrupciﾃｳn breve: El proceso de migraciﾃｳn se detiene abruptamente. Una vez que se restaura la conectividad, el proceso se reanuda y puede completarse con ﾃｩxito, aunque con un retraso.
* Interrupciﾃｳn prolongada: El proceso de migraciﾃｳn se agota y falla. La mﾃ｡quina virtual de origen continﾃｺa funcionando normalmente en el nodo de origen.

El proceso de migraciﾃｳn se ejecuta en modo punto a punto, lo que significa que el daemon de libvirt (libvirtd) en el nodo de origen controla la migraciﾃｳn llamando directamente al daemon de destino. Un mecanismo de keepalive integrado asegura que la conexiﾃｳn del cliente permanezca activa durante el proceso de migraciﾃｳn. Si la conexiﾃｳn permanece inactiva durante un perﾃｭodo especﾃｭfico, se cierra y el proceso de migraciﾃｳn se aborta.

Por defecto, el intervalo de keepalive se establece en 5 segundos y el recuento de reintentos se establece en 5. Dado estos valores predeterminados, el proceso de migraciﾃｳn se aborta si la conexiﾃｳn estﾃ｡ inactiva durante 30 segundos. Sin embargo, la migraciﾃｳn puede fallar antes o despuﾃｩs, dependiendo de las condiciones reales del clﾃｺster.
